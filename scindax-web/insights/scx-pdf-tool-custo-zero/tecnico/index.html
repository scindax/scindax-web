🛠 SCX Na Prática

03 Jul 2026·Tempo de leitura: 14 minutos·Tutorial interativo

Implante um PDF Tool serverless com IA de borda — do zero, sem gastar um real

Público: desenvolvedores, arquitetos de software e profissionais técnicos que querem montar e publicar essa stack com as próprias mãos.

← Versão para gestores

Ferramentas de manipulação de PDF que rodam no servidor exigem armazenar o arquivo em algum lugar, escalar CPU para operações pesadas de renderização e assumir o risco de manter documentos potencialmente sensíveis em trânsito.

A restrição era dura: zero orçamento para servidor dedicado, e nenhuma equipe de infraestrutura disponível para manter um backend tradicional.

Este não é só um post de arquitetura. É um checklist executável: marque cada passo conforme for configurando a sua própria stack e, ao final, gere um selo com o seu nome.

Decisões de arquitetura

1. Processamento client-side com PDF.js + pdf-lib

Decisão: renderizar miniaturas com PDF.js e montar/editar o PDF final com pdf-lib, tudo no navegador. Cada arquivo é lido como ArrayBuffer; o PDF.js desenha a primeira página em um <canvas> para a miniatura, e o pdf-lib copia as páginas selecionadas para um novo documento, retornando um Uint8Array convertido em Blob para download.

Isso evita envio de binário para qualquer servidor e mantém a performance previsível — sem fila de processamento remoto.

2. API mínima em função de borda

Decisão: expor só duas rotas em um Worker: uma para checar cota de uso, outra para pedir sugestão de nome. GET /check-limit lê um contador no KV, indexado por IP e data. POST /suggest-name recebe até 500 caracteres de texto (nunca o arquivo) e retorna um JSON com o nome sugerido.

3. Modelo de IA com fallback em cadeia

Decisão: um modelo leve como opção primária e, se falhar, um modelo mais robusto como segunda tentativa; se os dois falharem, um nome genérico gerado no próprio Worker. O prompt de sistema é fixo e força resposta em JSON puro, que passa por sanitização (minúsculas, sem acento, snake_case, limite de 50 caracteres) antes de voltar ao cliente.

4. Cache por hash

Decisão: antes de chamar a IA, calcular um hash do texto recebido e checar o KV. Chave no formato suggestion:{hash}, TTL de alguns dias. Um cache hit custa uma leitura de KV, não uma inferência.

5. Validação humana sem atrito visível

Decisão: um widget de bot-protection invisível antes de aceitar qualquer chamada às rotas da API. Um token é gerado em background e enviado como header (X-Turnstile-Token) em cada requisição; falha na validação retorna 403 sem expor detalhes internos.

Como as partes se conectam

Arquivo lido como ArrayBuffer no navegador
        ↓
PDF.js renderiza miniaturas (lazy loading)
        ↓
Reordenação via SortableJS, seleção via checkbox
        ↓
Extração de texto da 1ª página (limite: 500 caracteres)
        ↓
GET /check-limit → valida token de bot + consulta contador no KV
        ↓
POST /suggest-name → calcula hash(texto) → checa cache no KV
        ↓
Cache miss → chama modelo primário → fallback em cadeia se falhar
        ↓
Resposta sanitizada, salva no KV com TTL, contador incrementado
        ↓
pdf-lib monta o PDF final a partir do buffer original
        ↓
Blob gerado → download via elemento <a download>

O número que resume a decisão

A cota gratuita de inferência do modelo primário viabiliza, na ordem de milhares de chamadas por dia antes de qualquer custo aparecer — suficiente para cobrir o tráfego de uma ferramenta em fase de validação sem revisar orçamento.

Lições técnicas

1. Prompt determinístico reduz custo de parsing. Forçar o modelo a responder em JSON fixo elimina heurísticas de extração de texto — menos código, menos casos de borda.

2. Hash como chave de cache é mais barato do que parece. Uma função de hash simples sobre o texto de entrada já resolve boa parte da repetição de chamadas.

3. Fallback em cadeia é mais barato do que redundância de infraestrutura. Um segundo modelo, mais um valor padrão gerado localmente, cobrem quase todos os cenários de falha sem exigir um segundo provedor.

4. Bindings com nomes parecidos escondem produtos diferentes. "AI", "AI Search" e "AI Search Namespace" não são a mesma coisa. Confira sempre a documentação do tipo exato de binding antes de assumir que está tudo certo.

5. Placeholder sem checklist de substituição vira bug de produção. Documentar exatamente onde cada valor sensível entra, e validar antes do deploy, evita o erro mais comum desse tipo de stack: publicar com uma secret ou uma URL ainda apontando para um valor de exemplo.
