Build travado em "Queued" no Cloudflare Pages: checklist de diagnóstico técnico

Público: desenvolvedores, analistas de infraestrutura e profissionais técnicos que mantêm pipelines de deploy contínuo baseados em Git.

← Versão para gestores

Um push para main foi confirmado no repositório. Minutos depois, a página em produção continuava servindo a versão anterior, reproduzido em múltiplos dispositivos e em aba anônima, eliminando cache de navegador como causa.

Sem acesso ao dashboard da hospedagem no momento da investigação, o diagnóstico precisou ser feito inteiramente a partir de fora: API pública do Git host, cabeçalhos HTTP e renderização real da página.

Este post documenta o checklist usado para isolar a falha camada por camada, até identificar que o problema estava na fila de execução de build do Cloudflare Pages, não no código, não na integração, não no cache.

Checklist de diagnóstico

1. API do Git host como fonte de verdade

Decisão: validar o estado do repositório remoto via API pública, independente do estado do Git local.

Motivos: o git status local e o git log só confirmam o que a máquina local enxerga, não garantem que o remoto reflete o mesmo estado nem informam o timestamp real do push. A API pública do GitHub responde isso sem depender de sessão autenticada nem de cache local.

Como funciona na prática:

curl -s https://api.github.com/repos/{owner}/{repo}/commits/{branch} \
  | grep -E '"sha"|"date"'

O retorno confirmou o SHA do commit mais recente na branch de produção e o timestamp exato do push, eliminando a hipótese de falha no envio.

2. Cabeçalhos HTTP para separar cache de ausência real

Decisão: inspecionar status code e Cache-Control / CF-Cache-Status em vez de inferir o estado a partir da renderização.

Motivos: um 404 genuíno e um 200 servido do cache de edge têm sintomas visuais parecidos (conteúdo "errado"), mas causas completamente diferentes. A resposta HTTP distingue os dois imediatamente.

curl -sI https://dominio.com/rota-nova/
# HTTP/1.1 404 Not Found  → rota nunca chegou ao servidor

curl -sI https://dominio.com/rota-existente/
# HTTP/1.1 200 OK
# CF-Cache-Status: HIT    → servido do edge, possivelmente desatualizado

A rota nova retornou 404 real (a build nunca publicou aquele arquivo). A rota existente retornou 200 com CF-Cache-Status: HIT, mesmo com Cache-Control: public, max-age=0, must-revalidate, indicando que o conteúdo servido não era o mais recente disponível na origem.

3. Renderização real via headless browser

Decisão: carregar a página em um navegador headless e capturar o DOM pós-JavaScript, em vez de inspecionar apenas o HTML bruto retornado pelo curl.

Motivos: páginas com conteúdo injetado via JS podem divergir do HTML inicial; uma captura visual real elimina falsos positivos e reproduz exatamente o que o usuário final vê.

chrome --headless=new --disable-gpu --no-sandbox \
  --window-size=1400,1200 \
  --screenshot=out.png \
  https://dominio.com/rota/

O screenshot confirmou visualmente o que os cabeçalhos já indicavam: a versão publicada em produção não incluía as mudanças do commit mais recente.

4. Checklist de integração Git → hospedagem

Decisão: validar, nesta ordem, (1) status da conexão do repositório, (2) branch de produção configurada, (3) escopo de acesso do GitHub App ao repositório.

Motivos: essas três causas cobrem a maioria das falhas silenciosas de CI/CD baseado em Git App (em vez de webhook clássico) e são verificáveis diretamente no painel, sem precisar de acesso a logs de build.

Como funciona na prática: as três checagens retornaram normais, repositório conectado, branch de produção correta (main), e o GitHub App com acesso válido ao repositório. Isso eliminou a camada de integração como causa.

5. Deploy manual como teste controlado

Decisão: disparar manualmente um novo deployment (via "Retry deployment" / "Create deployment") e tratar o resultado como dado de diagnóstico, não como correção.

Motivos: se o deploy manual completar, isola a falha no gatilho automático (webhook). Se travar do mesmo jeito, isola a falha na etapa de execução do build, descartando a hipótese de webhook quebrado.

Como funciona na prática: o deploy manual permaneceu em status Queued indefinidamente, sem transicionar para Building e sem gerar log de erro, apontando para a fila de execução, não para o gatilho.

Fluxo da pipeline e onde travou

git push origin main
        ↓
GitHub registra o commit (confirmado via API)
        ↓
Webhook/GitHub App notifica o Cloudflare Pages
        ↓
Deployment criado com status Queued
        ↓
        ✕  [trava aqui, nunca transiciona para Building]
        ↓
Build executa
        ↓
Deploy publicado
        ↓
Cache do edge invalidado

Impacto

Tempo entre push e confirmação na API do GitHub: menos de 1 minuto. Tempo em Queued sem transicionar para Building: mais de 40 minutos, sem log de erro, sem timeout, sem notificação de falha.

Lições aprendidas

1. Trate a pipeline como uma cadeia de estados observáveis.
Cada etapa, como push, webhook, fila, build, deploy ou invalidação de cache, deveria ser inspecionável isoladamente. Quando não é, o diagnóstico vira adivinhação.

2. Cache de CDN é um estado esperado, não um bug.
CF-Cache-Status: HIT com conteúdo desatualizado é o comportamento normal de servir do edge. O diagnóstico da causa raiz precisa vir de outra camada, não do cache em si.

3. Prefira fontes autoritativas ao estado local.
Comparar o Git local com a API pública do host remoto elimina hipóteses de sincronização antes de suspeitar da hospedagem.

4. Ausência de erro também é um sinal.
Um status que não transiciona (Queued parado) é, por si, um dado de diagnóstico. Aponta para bloqueio na camada de execução (ex: concorrência de build no plano, build anterior preso), não na de configuração.

Notas técnicas adicionais

  • No plano gratuito do Cloudflare Pages, builds rodam com concorrência limitada. Um build anterior preso em execução pode bloquear a fila para os seguintes.
  • Antes de reconectar a integração Git do zero, verifique se há deployments anteriores presos em Building e cancele-os. Isso libera a fila sem precisar reautorizar o GitHub App.
  • CF-Cache-Status: HIT não invalida a hipótese de deploy ausente. Sempre cruze com o status HTTP de uma rota nova, que não pode estar em cache por definição.
  • Para builds que ficam em Queued sem causa aparente na configuração, vale checar o status da plataforma antes de reautorizar integrações.
