Construindo uma infraestrutura serverless com Cloudflare: a arquitetura da Scindax

Público: desenvolvedores, arquitetos de software, CTOs e profissionais técnicos que querem entender como construir uma stack serverless completa com baixo custo.

← Versão para gestores

O desafio era construir uma infraestrutura completa: site institucional, captura de leads via formulário, API própria, banco de dados relacional e um sistema de gestão interno — tudo com orçamento inicial zero e preparado para escalar.

A abordagem tradicional (contratar servidores, configurar VPS, gerenciar bancos de dados, lidar com SSL e CDN) demandaria investimento e manutenção constantes.

Optamos por uma arquitetura serverless-first, baseada em serviços com planos gratuitos generosos e escalabilidade horizontal. Este post detalha cada escolha técnica.

Decisões técnicas

1. Frontend: Cloudflare Pages + GitHub

Decisão: Cloudflare Pages para hospedagem, integrado ao GitHub para deploy contínuo.

Motivos:

  • Deploy automático em cada push para main.
  • HTTPS gerenciado automaticamente.
  • CDN global sem configuração adicional.

Exemplo de estrutura:

scindax-web/
├── index.html
├── css/
│   └── styles.css
├── js/
│   ├── particles.js
│   ├── navigation.js
│   └── main.js
├── assets/
├── robots.txt
└── sitemap.xml

2. Formulário: Tally + Webhook

Decisão: Tally para criação do formulário, integração via webhook para nossa API.

Motivos:

  • Criação rápida de formulários complexos.
  • Webhooks nativos disparam requisições HTTP para nosso Worker.
  • Exportação de PDF integrada.

Payload recebido no Worker:

{
  "eventType": "FORM_RESPONSE",
  "data": {
    "submissionId": "OQYqDGp",
    "fields": [
      {
        "label": "Nome completo",
        "type": "INPUT_TEXT",
        "value": "João Silva"
      },
      {
        "label": "E-mail",
        "type": "INPUT_EMAIL",
        "value": "joao@empresa.com"
      }
    ]
  }
}

3. API Gateway: Cloudflare Worker

Decisão: Cloudflare Worker para construir a API REST, rodando no edge.

Motivos:

  • Execução no edge: baixa latência.
  • Integração nativa com D1.
  • 100.000 requisições/dia gratuitas.

Estrutura do Worker (resumida):

export default {
  async fetch(request, env) {
    const url = new URL(request.url);

    const corsHeaders = {
      "Access-Control-Allow-Origin": "*",
      "Access-Control-Allow-Headers": "Content-Type, x-api-key",
      "Access-Control-Allow-Methods": "GET, POST, OPTIONS",
    };

    // Preflight
    if (request.method === "OPTIONS") {
      return new Response(null, { headers: corsHeaders });
    }

    // Webhook do Tally
    if (request.method === "POST" && url.pathname === "/webhook/tally") {
      const payload = await request.json();
      const id = crypto.randomUUID();

      await env.DB.prepare(`
        INSERT INTO assessments (id, created_at, company_name, respondent_name, email, answers_json, tally_submission_id)
        VALUES (?, ?, ?, ?, ?, ?, ?)
      `).bind(
        id,
        payload.data.createdAt,
        payload.data.fields.find(f => f.label === "Nome da empresa")?.value ?? null,
        payload.data.fields.find(f => f.label === "Nome completo")?.value ?? null,
        payload.data.fields.find(f => f.type === "INPUT_EMAIL")?.value ?? null,
        JSON.stringify(payload),
        payload.data.submissionId
      ).run();

      return Response.json({ success: true, id }, { headers: corsHeaders });
    }

    // Listagem protegida por API Key
    if (request.method === "GET" && url.pathname === "/api/assessments") {
      const apiKey = request.headers.get("x-api-key");
      if (apiKey !== env.API_KEY) {
        return new Response("Unauthorized", { status: 401, headers: corsHeaders });
      }

      const { results } = await env.DB.prepare(`
        SELECT * FROM assessments ORDER BY created_at DESC
      `).all();

      return Response.json(results, { headers: corsHeaders });
    }

    return new Response("Not Found", { status: 404, headers: corsHeaders });
  },
};

Autenticação: API Key armazenada como Secret na Cloudflare, acessada via env.API_KEY.

4. Banco de Dados: Cloudflare D1

Decisão: Cloudflare D1 (SQLite distribuído).

Motivos:

  • Integração nativa com Workers (bindings diretos).
  • SQL completo com índices e transações.
  • Plano gratuito: 5 GB de armazenamento, 5 milhões de leituras/mês.

Schema:

CREATE TABLE assessments (
    id TEXT PRIMARY KEY,
    created_at TEXT NOT NULL,
    company_name TEXT,
    respondent_name TEXT,
    email TEXT,
    phone TEXT,
    tally_submission_id TEXT,
    answers_json TEXT NOT NULL,
    status TEXT DEFAULT 'pending',
    source TEXT DEFAULT 'tally',
    metadata_json TEXT
);

CREATE INDEX idx_assessments_created_at ON assessments(created_at DESC);

Binding no wrangler.toml:

[[d1_databases]]
binding = "DB"
database_name = "scindax-ames"
database_id = "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"

5. Sistema de Gestão

Decisão: desenvolvemos um dashboard interno em React.

Integração com a API:

const API_BASE = process.env.AMES_API_URL;
const API_KEY = process.env.AMES_API_KEY;

export const amesApi = {
  async getAssessments() {
    const response = await fetch(`${API_BASE}/api/assessments`, {
      headers: { 'x-api-key': API_KEY },
    });
    if (!response.ok) throw new Error(`Erro: ${response.status}`);
    return response.json();
  },
};

Tratamento de erros: 401, 404, 500 e timeout são tratados com mensagens amigáveis.

Fluxo arquitetural completo

[Usuário] → [Tally] → [Webhook] → [Worker API] → [D1 Database]
                                           ↓
[Manager] → [GET /api/assessments] → [Worker API] ← [D1 Database]

Impacto

Custo mensal da infraestrutura nos primeiros meses: menos de R$ 50.

Os recursos permanecem majoritariamente no plano gratuito. O custo só aumenta quando a demanda cresce — e a arquitetura escala horizontalmente sem migrações.

Lições aprendidas

1. Serverless-first reduz custo e complexidade inicial.
Elimina a necessidade de gerenciar servidores, escalabilidade e disponibilidade. A plataforma cuida disso.

2. Integração nativa entre serviços agiliza o desenvolvimento.
Workers + D1 eliminam camadas de conexão. Não precisamos de ORMs nem drivers adicionais.

3. Segurança deve ser nativa, não um complemento.
API Key + CORS configurados corretamente já resolvem a maior parte dos riscos de exposição.

4. Tenha um plano de migração desde o início.
Usamos Google Sheets no MVP, mas a estrutura do banco já estava desenhada para substituí-lo. A migração foi planejada e executada sem perda de dados.

Notas técnicas adicionais

  • Versionamento de código: o Worker está versionado no Cloudflare, permitindo rollback.
  • Ambientes: podemos ter versões de desenvolvimento e produção.
  • Monitoramento: logs disponíveis no painel de Observabilidade do Cloudflare.
  • Performance: cada requisição ao Worker é processada em < 50ms.
